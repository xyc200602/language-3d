## 2026-06-08 - Task 78: 装配特征自动匹配 — 零件+连接方式自动生成特征

### What was done:
- 新建 `src/lang3d/tools/assembly_matcher.py` — 装配特征自动匹配系统
  - `FastenerSelection` 数据类：螺栓尺寸/长度、螺母/垫圈规格
  - `AssemblyMatchResult` 数据类：完整匹配结果（特征、紧固件、约束、验证）
  - `AssemblyMatcher` 类：5步自动匹配流程
    1. `_identify_functional_part()`: 通过目录ID或类别识别功能件/结构件
    2. `_generate_interface_features()`: 从 MountingInterface 生成配合特征（优先路径）
    3. `_generate_heuristic_features()`: 使用 ConnectionFeatureEngine 启发式生成（回退路径）
    4. `_select_fasteners()`: 自动选择螺栓长度+螺母+垫圈
    5. `_setup_constraints()`: 根据连接方式设置配对约束
    6. `_verify_match()`: 验证匹配结果合理性
  - `assembly_match_tool_factory()`: 注册为 assembly_auto_match 工具
  - `_opposite_face()` 辅助函数
- 修改 `agent/core.py` — 注册 assembly_auto_match 工具
- 修改 `tools/base.py` — 新增 assembly_auto_match 到 TOOL_CATEGORIES
- Bug fix: `_select_fasteners()` 中当两零件都是结构件时 `func_part`/`struct_part` 为 None 的 AttributeError

### Testing:

## 2026-06-08 - Task 77: 装配约束求解升级 — 从 anchor-face 到配对约束

### What was done:
- 新建 `src/lang3d/tools/mating_constraint.py` — 完整的配对约束求解系统
  - `MatingConstraint` 数据类：6种约束类型（coincident/concentric/distance/angle/parallel/perpendicular）
  - `SolvedPosition` 数据类：position + rotation_matrix + axis-angle
  - `ConstraintSolver` 类：输入 Assembly + 可选 MatingConstraints → 输出每个零件 position + rotation
  - `anchor_to_constraints()`: 将旧 anchor-face Joint 自动转换为 MatingConstraint（向后兼容）
  - 6种约束求解器:
    - `_solve_coincident()`: 面贴合，法向量反向对齐
    - `_solve_concentric()`: 轴孔同心，轴心线对齐
    - `_solve_distance()`: 保持指定偏移距离
    - `_solve_angle()`: 保持指定角度
    - `_solve_parallel()`: 平行约束
    - `_solve_perpendicular()`: 垂直约束（90°旋转）
  - `_alignment_rotation()`: 向量对齐旋转计算（含平行/反平行/一般情况）
  - `_build_constraint_graph()` + BFS顺序求解（从基准件开始）
  - `constraint_solve_tool_factory()`: 注册为 constraint_solve 工具
- 修改 `agent/core.py` — 注册 ConstraintSolver 工具
- 修改 `tools/base.py` — 新增 constraint_solve 到 TOOL_CATEGORIES
- 完整的向量/矩阵运算库（不依赖 numpy）

### Testing:
- 新建 tests/test_mating_constraint.py — 32 个测试
- 全部 606 个相关测试通过
- 零回归

### Notes:
- 向后兼容：不带显式 constraints 时，自动从 Joint 的 anchor-face 转换
- 约束求解采用 BFS 顺序，从根节点逐步向外

### 装配求解器问题分析

**根因**: `_compute_child_transform()` 只计算 anchor 面的中心点，不处理同面多子零件的水平分布。
- 4 轮全部在 base_plate bottom 面中心 (0, 0, -Z)
- 4 铜柱全部在 base_plate top 面中心 (0, 0, +Z)
- 双臂全部在 top_plate top 面中心 (0, 0, +Z)
- 结果：41 零件堆成一根竖直棍子

**修复方案**: 新增 `_face_extent_for_part`、`_tangent_vectors_for_anchor`、`_compute_distribution_offset` 三个函数，按 `(parent, parent_anchor)` 分组子零件，根据数量自动选择分布模式（2→line, 3-4→grid 四角, 5+→circle）。单子零件行为不变，向后兼容。

### VLM 装配验证分析

**现状**: VLM 只验证单个零件几何（cad_verify），从未验证整体装配布局。
**方案**: 新增 AssemblyVisualVerifier → 多角度截图 → VLM 分析 → 检测垂直堆叠等问题 → 自动修正 offset → 重新验证（最多 3 轮）。
**复用**: cad_verify 的多角度截图、JSON 解析、置信度投票机制。
**预计**: 每轮验证 ~30s，3 轮 ~90s，token 成本可忽略。

#### TaskDAG 依赖图 (`agent/dag.py` 新建)

```
TaskDAG
  ├─ add_step(step, dependencies, agent_role) → DAGNode
  ├─ get_ready_steps() → list[DAGNode]          # 所有依赖已完成的步骤
  ├─ mark_completed(step_id) → list[str]         # 返回新解锁的 step IDs
  ├─ mark_failed(step_id) → list[str]            # 级联标记依赖者为 SKIPPED
  ├─ parallel_groups() → list[list[DAGNode]]     # Kahn 算法拓扑排序 → 波次分组
  ├─ _detect_cycles()                             # DFS 检测环 → ValueError
  ├─ to_dict() → dict                            # 序列化给 Web 面板
  └─ from_plan(plan, assembly) → TaskDAG          # 从 Plan + Assembly 构建
```

- `parallel_groups()` 使用 Kahn 算法进行拓扑排序，将节点分为并行波次
- `from_plan()` 可从 `Assembly.joints` 推断依赖关系（child 依赖 parent）
- `agent_role` 根据步骤的 `expected_tools` 自动推断（modeling/vision/gui/general）

### 视觉模型评估 + 多模型路由

**GLM 视觉模型全面评估**：

| 模型 | Coding Plan API | 标准 API | 备注 |
|------|----------------|----------|------|
| GLM-4V-Flash | 可用 | 可用 | 免费 |
| GLM-4V | 可用 | 可用 | 收费 |
| GLM-4V-Plus | 可用 | 可用 | 收费，质量最好 |
| GLM-4.6V | 返回空内容 | 可用 | 需标准 API |
| GLM-4.6V-Flash | 返回空内容 | 可用 | 需标准 API |
| GLM-5V-Turbo | 429 限权 | **可用** | 需标准 API |

**GLM-4.6V "返回空内容" 排查**：不是模型不能用，而是之前 `max_tokens=500` 不够。GLM-4.6V 输出以 `\n` 开头 + `finish_reason=length` 截断 = 看起来像空内容。实际上需要 `max_tokens >= 2048` 才能得到完整回答。

**max_tokens 上限测试**：

| 模型 | 实际 API 上限 | 最佳设置 |
|------|-------------|---------|
| GLM-4V-Flash | 1024 | 1024 |
| GLM-4V-Plus | 2048 | 2048 |
| GLM-4.6V-Flash | 16384+ | 16384 |
| GLM-4.6V | 16384+ | 16384 |

**视觉模型路由实现** — 4 级路由配置：

| Level | 模型 | max_tokens | 速度 | 适用场景 |
|-------|------|-----------|------|---------|
| `fast` | GLM-4V-Flash | 1024 | ~1s | 快速识别（颜色、形状） |
| `standard` | GLM-4V-Plus | 2048 | ~3s | 精准分析（UI 识别） |
| `detailed` | GLM-4.6V-Flash | **16384** | ~20s | CAD 验证（详细描述） |
| `maximum` | GLM-4.6V | **16384** | ~40s | 复杂检测（最全面） |

**改动的文件**：
- `models/base.py`：vision() 默认 max_tokens 2048→4096
- `models/glm.py`：默认 max_tokens 1024→4096；新增 `alt_client`（标准 API）；`VISION_MODEL_CAPS`
- `models/router.py`：新增 `VisionDetail` 枚举（4 级）；`VISION_DETAIL_MODELS` 路由表
- `tools/vlm.py`：4 个工具全部新增 `detail` 参数，默认 `detailed`

**Testing**: 38/38 单元测试通过；8/8 VLM 感知 e2e 测试通过；4/4 视觉路由级别验证通过

**关键技术发现**：
1. **双 API 端点策略**：Coding Plan API (`/coding/paas/v4`) 不支持 GLM-4.6V 系列和 GLM-5V-Turbo，需要 Standard API (`/api/paas/v4`)。GLMBackend 自动根据模型名选择端点
2. **max_tokens 是视觉模型可用性的关键**
3. **多模型路由的经济性**：简单任务用免费的 Flash，CAD 验证用详细的 4.6V-Flash
