#### Phase 2: VLM 验证准确率提升

**修改 `src/lang3d/tools/vlm.py`**：

验证提示词重设计（chain-of-thought）：
- 先描述看到的内容 → 再逐项对比 → 最后给出结论
- 允许 markdown code block 包裹 JSON（VLM 自然倾向）
- 移除冗长示例（节省 ~200 tokens）
- 新增 `confidence` 字段（high/medium/low）
- 添加"宽容匹配"指令：仅形状/结构明显不同才报 MISMATCH

JSON 解析三阶段策略（重写 `_parse_verification_json`）：
1. **策略 1**：从 ` ```json ... ``` ` markdown code block 提取
2. **策略 2**：括号深度追踪提取裸 JSON 对象（支持多行、嵌套）
3. **策略 3**：逐字段正则提取回退

新增 `_normalize_verification(data)` 辅助函数：
- 统一布尔值解析（true/True/yes/1）
- null/"null"/"None"/"" 统一为 "None"

**测试**：`tests/test_vlm_parsing.py` 15 tests 通过（原始 JSON、markdown 包裹、多行、纯文本回退、中文、混合大小写）

#### Phase 3: 智能自修复循环

**新建 `src/lang3d/agent/fix_strategy.py`**：
- `FailureType` 枚举：MISSING_FEATURE / WRONG_DIMENSION / WRONG_POSITION / WRONG_SHAPE / EXTRA_FEATURE / ASSEMBLY_ERROR / UNKNOWN
- `FixContext` 数据类：失败类型、描述、目标特征、期望/实际值、修复历史
- `classify_failure(cad_verify_result, expected)` → `FixContext`：关键词启发式分类
- `generate_fix_hint(ctx)` → `str`：根据失败类型生成中文定向修复提示
- `check_convergence(previous_fixes, current_result, threshold=0.8)` → `bool`：difflib 文本相似度检测修复循环

**修改 `src/lang3d/agent/core.py`**：
- 替换 `_run_direct()` 中朴素 auto-fix 逻辑：
  - 用 `classify_failure()` 分类失败类型
  - 用 `generate_fix_hint()` 生成定向提示
  - 用 `check_convergence()` 检测循环 → 提示"尝试完全不同的建模方法"

**修改 `src/lang3d/agent/reflector.py`**：
- 工具历史窗口 5 → 10
- 单条结果截断 200 → 300 字符

**问题背景**：工作区 60 个模型文件中 34 个是 .FCStd，2 个是 .STEP，浏览器无法直接渲染（用户反馈"3d 渲染大部分都渲染不出来"）。

**STEP → STL 后端转换** (`src/lang3d/web/app.py` 新增 `/api/convert-step`):
- FreeCADCmd (headless) 执行转换脚本：`Part.Shape().read(step)` → `MeshPart.meshFromShape(shape, LinearDeflection=0.5, AngularDeflection=0.523599)` → `mesh.write(stl)`
- 缓存为 `<name>.preview.stl`，如缓存比源文件新则复用
- 60秒超时，路径安全验证

**FCStd → STL 后端转换** (`src/lang3d/web/app.py` 新增 `/api/convert-fcstd`):
- `FreeCAD.openDocument(fcstd)` 打开文档
- 遍历 `doc.Objects`，跳过 `Visibility=False` 的对象
- 收集所有非空 `obj.Shape` → `Part.makeCompound(shapes)` 合并
- `MeshPart.meshFromShape(compound)` 网格化 → `mesh.write(stl)` 输出
- 120秒超时（FCStd 可能包含复杂多体模型）

**`/api/models` 排序与元数据增强**：
- 新增字段：`previewable`（STL/OBJ 且非空）、`convertible`（STEP/FCStd 且 FreeCAD 可用）、`empty`（0 字节）
- 排序逻辑：`(rank, -mtime)` — previewable(rank=0) → convertible(rank=1) → blocked(rank=2)
- 返回 `freecad_available` 标志

**前端 badge 与交互** (`src/lang3d/web/static/index.html`)：
- 绿色 `STL` badge：浏览器直接渲染
- 橙色 `STEP→STL` / `FCStd→STL` badge：点击后调用转换端点
- 灰色 `EMPTY` / `FCSTD` 等 badge + disabled 状态：不可点击
- `openModel()` 根据 `model.ext` 路由到 `/api/convert-step` 或 `/api/convert-fcstd`

**Testing**:
- `pytest tests/test_web_panel.py -v`：32 passed（无回归）
- 手动验证：base_plate.step → 9KB STL ✓；agent_cube_test.FCStd → 684B STL ✓；shoulder_link.FCStd → 78KB STL（多 shape 合并）✓；test_box.FCStd 缓存命中 ✓
- 浏览器手动测试：用户确认 "stl 都可以"

**Notes**:
- LinearDeflection=0.5 / AngularDeflection=0.523599（约30°）为预览级精度
- FCStd 中被隐藏的对象（`Visibility=False`）会被跳过
- `Part.makeCompound(shapes)` 合并多个 shape 后再 tessellate
- `with_suffix(".preview.stl")` 会替换原后缀，`foo.FCStd` 和 `foo.step` 都生成 `foo.preview.stl`（同名时后者覆盖前者，不影响功能）
- FreeCAD 转换是同步阻塞的（subprocess.run），高并发场景可能需要改为异步队列

#### TaskDAG 依赖图 (`agent/dag.py` 新建)

```
TaskDAG
  ├─ add_step(step, dependencies, agent_role) → DAGNode
  ├─ get_ready_steps() → list[DAGNode]          # 所有依赖已完成的步骤
  ├─ mark_completed(step_id) → list[str]         # 返回新解锁的 step IDs
  ├─ mark_failed(step_id) → list[str]            # 级联标记依赖者为 SKIPPED
  ├─ parallel_groups() → list[list[DAGNode]]     # Kahn 算法拓扑排序 → 波次分组
  ├─ _detect_cycles()                             # DFS 检测环 → ValueError
  ├─ to_dict() → dict                            # 序列化给 Web 面板
  └─ from_plan(plan, assembly) → TaskDAG          # 从 Plan + Assembly 构建
```

- `parallel_groups()` 使用 Kahn 算法进行拓扑排序，将节点分为并行波次
- `from_plan()` 可从 `Assembly.joints` 推断依赖关系（child 依赖 parent）
- `agent_role` 根据步骤的 `expected_tools` 自动推断（modeling/vision/gui/general）

### 视觉模型评估 + 多模型路由

**GLM 视觉模型全面评估**：

| 模型 | Coding Plan API | 标准 API | 备注 |
|------|----------------|----------|------|
| GLM-4V-Flash | 可用 | 可用 | 免费 |
| GLM-4V | 可用 | 可用 | 收费 |
| GLM-4V-Plus | 可用 | 可用 | 收费，质量最好 |
| GLM-4.6V | 返回空内容 | 可用 | 需标准 API |
| GLM-4.6V-Flash | 返回空内容 | 可用 | 需标准 API |
| GLM-5V-Turbo | 429 限权 | **可用** | 需标准 API |

**GLM-4.6V "返回空内容" 排查**：不是模型不能用，而是之前 `max_tokens=500` 不够。GLM-4.6V 输出以 `\n` 开头 + `finish_reason=length` 截断 = 看起来像空内容。实际上需要 `max_tokens >= 2048` 才能得到完整回答。

**max_tokens 上限测试**：

| 模型 | 实际 API 上限 | 最佳设置 |
|------|-------------|---------|
| GLM-4V-Flash | 1024 | 1024 |
| GLM-4V-Plus | 2048 | 2048 |
| GLM-4.6V-Flash | 16384+ | 16384 |
| GLM-4.6V | 16384+ | 16384 |

**视觉模型路由实现** — 4 级路由配置：

| Level | 模型 | max_tokens | 速度 | 适用场景 |
|-------|------|-----------|------|---------|
| `fast` | GLM-4V-Flash | 1024 | ~1s | 快速识别（颜色、形状） |
| `standard` | GLM-4V-Plus | 2048 | ~3s | 精准分析（UI 识别） |
| `detailed` | GLM-4.6V-Flash | **16384** | ~20s | CAD 验证（详细描述） |
| `maximum` | GLM-4.6V | **16384** | ~40s | 复杂检测（最全面） |

**改动的文件**：
- `models/base.py`：vision() 默认 max_tokens 2048→4096
- `models/glm.py`：默认 max_tokens 1024→4096；新增 `alt_client`（标准 API）；`VISION_MODEL_CAPS`
- `models/router.py`：新增 `VisionDetail` 枚举（4 级）；`VISION_DETAIL_MODELS` 路由表
- `tools/vlm.py`：4 个工具全部新增 `detail` 参数，默认 `detailed`

**Testing**: 38/38 单元测试通过；8/8 VLM 感知 e2e 测试通过；4/4 视觉路由级别验证通过

**关键技术发现**：
1. **双 API 端点策略**：Coding Plan API (`/coding/paas/v4`) 不支持 GLM-4.6V 系列和 GLM-5V-Turbo，需要 Standard API (`/api/paas/v4`)。GLMBackend 自动根据模型名选择端点
2. **max_tokens 是视觉模型可用性的关键**
3. **多模型路由的经济性**：简单任务用免费的 Flash，CAD 验证用详细的 4.6V-Flash
