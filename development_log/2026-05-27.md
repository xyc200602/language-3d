### 遇到的问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| `pyproject.toml` build-backend 报错 | 生成时用了 `setuptools.backends._legacy:_Backend` | 改为 `setuptools.build_meta` |
| `test_file_edit` 失败 | 测试中未注册 `FileReadTool` | 添加 `registry.register(FileReadTool())` |
| GLM API 返回空内容 | `max_tokens=20` 太小，全被 reasoning_tokens 消耗 | 增大到 512+，GLM-5.1 是推理模型需要更多 token 预算 |
| GLM 后端用了错误的 SDK | 初始按 Anthropic 兼容协议编写 | 重写为 OpenAI SDK，Coding Plan 使用 OpenAI 兼容协议 |

---

### 关键技术发现

1. **GLM-5.1 是推理模型**：API 返回 `reasoning_tokens` 字段，实际输出 token = reasoning_tokens + 可见文本 tokens。调用时 `max_tokens` 需要留足余量（建议 >= 1024）。
2. **Coding Plan 端点完全兼容 OpenAI SDK**：`client.chat.completions.create()`、tools/function calling 格式均与 OpenAI 一致。
3. **工具调用格式**：`finish_reason: "tool_calls"`，参数在 `choices[0].message.tool_calls[].function.arguments`（JSON 字符串，需 parse）。

#### 3. FreeCAD 工具模块 (`src/lang3d/tools/freecad.py`)

- 工具层将建模操作序列化为 JSON

- 通过 `_run_freecad_script()` 调用 FreeCAD 自带的 Python 执行

| `fc_plate_with_holes` | 带安装孔的矩形板 |

### 遇到的问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| `import FreeCAD` 报 DLL 冲突 | FreeCAD 内置 Python 3.11 vs 系统 Python 3.12 | 改用子进程桥接：调用 FreeCAD 自带 Python 执行 |

#### 2. `ToolRegistry` 参数名冲突修复

- **问题**：`ToolRegistry.execute(name, **kwargs)` 的第一个参数 `name` 与 FreeCAD 工具的 `name` 参数冲突
- **现象**：Agent 调用 `fc_new_doc(name="Cube")` 时变成 `execute("fc_new_doc", name="Cube")`，`name` 参数被传了两次
- **修复**：将 `ToolRegistry.execute()` 的参数名从 `name` 改为 `tool_name`

#### 3. `fc_batch` 工具

新增 `fc_batch` 工具，解决 FreeCAD 子进程无状态的核心问题：

- **问题**：每个 `fc_*` 工具调用创建独立的 FreeCAD 子进程，操作之间没有状态持久化。Agent 调用 `fc_make_box` 创建的立方体，在 `fc_boolean` 时已经不存在了。
- **方案**：`fc_batch` 接受一个操作列表，在同一个 FreeCAD 进程中按顺序执行所有操作
- Agent 可以用一次 `fc_batch` 调用完成：new_doc → make_box → make_cylinder → move → boolean → export_stl

#### 4. Agent 端到端测试 (`tests/test_agent_e2e.py`)

**这是框架的真正验证**：不直接调用任何工具函数，而是通过 Agent 循环让 GLM-5.1 自主完成建模。

**测试流程**：
```
用户自然语言任务 → GLM-5.1 分析 → 自主选择工具和参数 → 执行 → 验证
```

**Task 1 — 立方体+贯穿孔（25.3s）**：

Agent 自主选择了 `fc_batch`，一次性组织了 6 步操作：
1. `new_doc` → 创建文档
2. `make_box` → 30x30x30mm 立方体
3. `make_cylinder` → R5 贯穿孔圆柱
4. `move` → 移动到中心 (15,15,0)
5. `boolean cut` → 立方体减圆柱
6. `export_stl` → 导出

然后 Agent 还自主调用了 `mesh_stats` 验证网格质量（体积 24,645 mm³, 520 面），确认贯穿孔正确（体积从 27,000 减少了约 2,355 mm³）。

**Task 2 — 套筒零件（16.2s）**：

Agent 直接选择了 `cylinder_with_hole` 高级模式 + `export_stl`，一步到位。然后用 `mesh_stats` 验证尺寸匹配。

---

### 遇到的问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| `ToolRegistry.execute()` 参数冲突 | 工具参数 `name` 与注册表方法参数 `name` 重名 | 方法参数改名为 `tool_name` |
| FreeCAD 子进程无状态 | 每次 `fc_*` 调用创建新进程，对象不持久化 | 新增 `fc_batch` 工具支持多步操作在单进程内执行 |
| `fc_chamfer` 在复杂形状上报 OCCError | OpenCASCADE 对合并后非流形边的倒角失败 | 移除复杂形状的倒角操作 |
| 控制台 GBK 编码报错 | Agent 返回含 emoji 的中文内容 | 测试脚本中用 ASCII 替换非兼容字符 |

---

### 关键技术发现

1. **GLM-5.1 具备工具编排能力**：能自主理解建模任务，选择正确的工具组合，组织参数，完成多步骤建模流程
2. **`fc_batch` 是 Agent 使用 FreeCAD 的正确方式**：单次调用执行完整建模流程，避免子进程无状态问题
3. **Agent 自主验证能力**：建模完成后自动调用 `mesh_stats` 检查网格质量，甚至自动安装了 `trimesh` 依赖
4. **Agent 自我修复能力**：首次测试时没有 `fc_batch`，Agent 发现 `fc_move` 失败后，自主改用 `fc_script` 写完整脚本绕过问题
5. **布尔运算+圆角是 OCC 的常见失败点**：FreeCAD 底层 OpenCASCADE 对非流形几何的 fillet/chamfer 操作不稳定

## Day 1 (续4): 视觉模型评估 + 多模型路由

### 概述

对 GLM 全系列视觉模型进行了系统评估，发现 GLM-4.6V 系列之前误判为"不可用"实际上是 max_tokens 过低导致截断。实现了多级视觉模型路由系统，Agent 可根据任务复杂度自动选择不同能力的视觉模型。

---

### GLM 视觉模型全面评估

#### 可用性探测

| 模型 | Coding Plan API | 标准 API | 备注 |
|------|----------------|----------|------|
| GLM-4V-Flash | 可用 | 可用 | 免费 |
| GLM-4V | 可用 | 可用 | 收费 |
| GLM-4V-Plus | 可用 | 可用 | 收费，质量最好 |
| GLM-4.6V | 返回空内容 | 可用 | 需标准 API |
| GLM-4.6V-Flash | 返回空内容 | 可用 | 需标准 API |
| GLM-5V-Turbo | 429 限权 | **可用** | 需标准 API |
| GLM-5V | 不存在 | 不存在 | - |

#### GLM-4.6V "返回空内容" 排查

**原因**：不是模型不能用，而是之前 `max_tokens=500` 不够。

- GLM-4.6V 输出以 `\n` 开头 + `finish_reason=length` 截断 = 看起来像空内容
- 实际上需要 `max_tokens >= 2048` 才能得到完整回答
- 在 `max_tokens=4096` 时，GLM-4.6V 能给出 1700-2100 tokens 的详细分析

#### max_tokens 上限测试

| 模型 | 实际 API 上限 | 最佳设置 |
|------|-------------|---------|
| GLM-4V-Flash | 1024 | 1024 |
| GLM-4V-Plus | 2048（>2048 返回 400） | 2048 |
| GLM-4.6V-Flash | 16384+ | 16384 |
| GLM-4.6V | 16384+ | 16384 |

#### 质量对比（CAD 线框分析，max_tokens=4096+）

| 模型 | 速度 | 输出 tokens | 质量 |
|------|------|-----------|------|
| GLM-4.6V | 47s | 2048 | 非常详细，但把 R5 同时识别为孔径+圆角（过度分析） |
| GLM-4.6V-Flash | 27s | 886 | 详细且准确，正确识别立方体+中心孔+R5 圆角 |
| GLM-4V-Plus | 3.2s | 116 | 简洁精准，正确识别所有特征 |
| GLM-4V-Flash | 3.3s | 202 | 基本正确 |

### 视觉模型路由实现

#### 架构

```
Agent 调用 VLM 工具（detail 参数）
  ↓
VisionDetail 枚举 → 选择模型 + max_tokens
  ↓
GLMBackend.vision(model=...) → 自动选择 API 端点
  ↓
Coding Plan API 或 Standard API
```

#### 4 级路由配置

| Level | 模型 | max_tokens | 速度 | 适用场景 |
|-------|------|-----------|------|---------|
| `fast` | GLM-4V-Flash | 1024 | ~1s | 快速识别（颜色、形状） |
| `standard` | GLM-4V-Plus | 2048 | ~3s | 精准分析（UI 识别） |
| `detailed` | GLM-4.6V-Flash | **16384** | ~20s | CAD 验证（详细描述） |
| `maximum` | GLM-4.6V | **16384** | ~40s | 复杂检测（最全面） |

#### 工具默认 detail 设置

| 工具 | 默认 detail | 理由 |
|------|-----------|------|
| `vlm_analyze` | `detailed` | 图片分析需要详细结果 |
| `screen_analyze` | `detailed` | 屏幕内容需要详细描述 |
| `window_analyze` | `detailed` | 窗口分析通常用于 CAD 场景 |
| `cad_verify` | `detailed` | CAD 验证需要详细对比 |

### 关键技术发现

1. **双 API 端点策略**：Coding Plan API (`/coding/paas/v4`) 不支持 GLM-4.6V 系列和 GLM-5V-Turbo，需要 Standard API (`/api/paas/v4`)。GLMBackend 现在自动根据模型名选择端点。
2. **max_tokens 是视觉模型可用性的关键**：GLM-4.6V 在低 max_tokens 下看起来像"返回空"，实际是截断 + `\n` 开头的组合效果。
3. **多模型路由的经济性**：简单任务用免费的 Flash（~1s），CAD 验证用详细的 4.6V-Flash（~20s），避免一刀切。
